## 1. 数据来源与 API 采集

### 1.1 工具链与环境

- **数据 API**：[FastF1](https://docs.fastf1.dev/)，Python 库，封装 F1 官方 Ergast API 及 FastF1 遥测数据服务
- **Python 依赖**：`fastf1`、`pandas`、`numpy`、`os`、`glob`、`pyreadr`（导出 `.rds`）
- **缓存目录**：`./cache/`（开启后同一 session 只下载一次，大幅缩短重跑时间）
- **输出目录**：`./output/`（所有 CSV 文件落盘位置）
- **编码约定**：所有 CSV 均以 `utf-8-sig` 编码存盘（兼容 Windows Excel 中文显示）

### 1.2 采集范围

| 维度 | 说明 |
|---|---|
| 赛季范围 | 2018–2026（`ALL_YEARS = range(2018, 2027)`） |
| 比赛类型 | 仅正式比赛站（`include_testing=False`，排除冬测） |
| 赛事筛选 | `RoundNumber > 0`（排除赛历中的非比赛行） |
| 2026 特殊说明 | 赛季进行中（截至 2026-03-27 已完成 Round 1–2），Round 3+ ⏳ 待补充 |

### 1.3 Session 类型说明

FastF1 中每个比赛周末包含多个 Session，本项目使用：

- **`R`（Race，正赛）**：用于采集表1（比赛结果）、表3（赛道状态）、表5（天气）
- **`Q`（Qualifying，排位赛）**：用于采集表2（排位赛最快圈）
  - ⚠️ Sprint 赛程（冲刺赛周末）同样使用 `Q`（主排位赛），**不使用 `SQ`（冲刺资格赛）**，保持数据口径一致

### 1.4 五张原始表总览

| 表号 | 文件名模式 | 采集 Cell | 所用 Session | 主要内容 |
|---|---|---|---|---|
| 表1 | `{year}_raw_race_results.csv` | Cell 3 / Cell 3b | R | 正赛逐车结果（成绩、积分、完赛状态、完成圈数） |
| 表2 | `{year}_raw_quali_laptimes.csv` | Cell 4 / Cell 3b | Q | 每车手排位赛最快圈时（秒），用于计算资源禀赋 |
| 表3 | `{year}_raw_track_status.csv` | Cell 3 / Cell 3b | R | 逐圈赛道状态字符串（SC/VSC/绿旗/红旗） |
| 表4 | `{year}_raw_event_meta.csv` | Cell 5 | 无需 load | 赛事静态元数据（赛道类型/长度/DRS 区数） |
| 表5 | `{year}_raw_weather_summary.csv` | Cell 3 / Cell 3b | R | 每场比赛天气摘要（雨战标记/温湿度/风速） |

> **全量合并文件**：`ALL_raw_{tname}.csv`，由 Cell 8（表1/2/3/5）和 Cell 5（表4）直接产出。

### 1.5 运行顺序（`1-other.ipynb`）

运行前确保已执行 `fastf1.Cache.enable_cache(cache_dir)`（Cell 2 中完成）。

1. **Cell 1**：导入所有依赖库（`fastf1`, `pandas`, `numpy`, `os`, `glob`）
2. **Cell 2**：配置参数
   - 设定 `ALL_YEARS`、`SKIP_YEARS`、缓存/输出目录
   - 定义 `CIRCUIT_META` 字典（43 条目，含赛道类型/长度/DRS 区数），以 **2023 赛季版本为基准**
3. **Cell 3**：R session 合并采集表1 + 表3 + 表5（范围：2018–2025）
   - **优化**：每站单次 `sess.load(laps=True, weather=True)` 同时获取圈速和天气，API 请求数降为原来的 1/3
4. **Cell 3b**：2026 赛季补充采集（独立 Cell，避免进行中赛历导致错误）
   - R session → 表1 + 表3 + 表5；Q session → 表2
   - `MAX_2026_ROUNDS = 2` 控制采集站数，赛季结束后调大重跑
5. **Cell 4**：Q session 采集表2 排位赛最快圈（范围：2018–2025）
6. **Cell 5**：采集表4 赛事元数据（无需 session load，仅用 `get_event_schedule()` + `CIRCUIT_META` 字典查表）
   - 此 Cell 直接产出 `ALL_raw_event_meta.csv`，无需 Cell 8 再合并
7. **Cell 8**：`glob` 匹配 `output/` 下所有 `{year}_{tname}.csv`，`pd.concat` 后存盘为 `ALL_{tname}.csv`（合并表1/2/3/5）

## 2. 超车事件采集与清洗

### 2.1 Notebook 说明

**文件**：`1-overtakes.ipynb`

**运行顺序**：Cell 1 → Cell A → Cell D → Cell E → Cell G → Cell H → Cell I → Cell J

> ⚠️ **必须按顺序运行**：Cell D 定义的 `_calc_*` 函数、Cell E 的 `SUPERSET_COLS`/`SEASON_SUPERSET_COLS`、Cell G 的 `process_one_race()` 函数均依赖前序 Cell 的全局变量和函数定义。

**采集范围**：
- 2018–2023, 2025 全量（Cell H 处理）
- 2024 已单独预采集（`SKIP_YEARS = {2024}`，新运行时跳过）
- 2026 仅 Round 1–2 ⏳（截至 2026-03-27）

### 2.2 四张输出表的层级关系

```
full_overtakes_season（所有候选超车事件）
        │
        ├── team_order_season
        │       筛选条件：IsTeammate == True
        │                AND TeamOrderConfidence ≥ 0.5
        │       含义：被判定为疑似车队指令的让位超车
        │
        ├── off_track_season
        │       筛选条件：(OffTrack_Flag==True OR Lockup_Flag==True OR LocalYellowFlag==True)
        │                AND OffTrackConfidence ≥ 0.5
        │       含义：被判定为越界/锁死/黄旗干扰导致的超车
        │
        └── clean_overtakes_season
                = full − {team_order ∪ off_track}
                去重键：(Round, LapNumber, Overtaker, Overtaken)
                含义：剔除干扰后的真实竞争性超车（核心分析用表）
```

### 2.3 可调判定阈值汇总

所有阈值在 **Cell A** 中定义，修改后重跑 Cell H 即可更新结果。

| 参数名 | 默认值 | 单位 | 含义 | 触发条件 |
|---|---|---|---|---|
| `TEAM_ORDER_THROTTLE_THRESHOLD` | `10.0` | % | Throttle_Drop_Index 上限 | `TDI > 10.0` 则贡献车队指令置信度 |
| `TEAM_ORDER_BRAKE_THRESHOLD` | `150.0` | ms | Brake_Pressure_Delta 上限 | `BPD > 150ms` 则贡献车队指令置信度 |
| `TEAM_ORDER_CONFIDENCE_THRESHOLD` | `0.5` | 0–1 | 标记为车队指令的最低置信度 | `TeamOrderConfidence ≥ 0.5` 则入 team_order 表 |
| `OFF_TRACK_VMIN_THRESHOLD` | `5.0` | % | Vmin_Deviation 上限 | `VminDev > 5.0%` 则贡献越界置信度 |
| `OFF_TRACK_CONFIDENCE_THRESHOLD` | `0.5` | 0–1 | 标记为越界干扰的最低置信度 | `OffTrackConfidence ≥ 0.5` 则入 off_track 表 |
| `TELEMETRY_HZ` | `10` | Hz | 遥测数据重采样频率 | 降低可加快处理速度，但特征精度下降 |

## 3. 核心变量精细对照

> 本节对应 Hallila, Frankort & Aversa (2024) AMJ 的核心变量，精确记录本项目中的实现方式与文献的映射关系。

---

### 3.1 `internal_overtakes` — 内部超车数

- **中文名**：内部（队内）超车数
- **单位**：次（整数）
- **分析层级**：车手-赛站（driver × round）
- **来源表**：`ALL_clean_overtakes_season.csv`
- **计算公式**：
  ```python
  df_clean[df_clean['IsTeammate'] == True] \
      .groupby(['Year', 'Round', 'Overtaker']) \
      .size() \
      .reset_index(name='internal_overtakes')
  ```
  即：在净超车表中，筛选 `IsTeammate == True`（超车者与被超者属于同一车队），按（年份、站次、车手）计数。
- **文献对应变量**：`Internal Competitive Actions`
- **文献原文定义**：
  > "We measure a rider's internal competitive actions as the number of internal overtakes they complete per race, i.e., the number of times they pass a teammate on the racetrack."
- **本项目与文献的差异**：
  - 文献为 MotoGP（摩托车），每队通常 2 名车手；F1 同为每队 2 名车手，结构一致
  - 文献使用 2004–2020 数据；本项目使用 F1 2018–2026

---

### 3.2 `external_overtakes` — 外部超车数

- **中文名**：外部（对他队）超车数
- **单位**：次（整数）
- **分析层级**：车手-赛站（driver × round）
- **来源表**：`ALL_clean_overtakes_season.csv`
- **计算公式**：
  ```python
  df_clean[df_clean['IsTeammate'] == False] \
      .groupby(['Year', 'Round', 'Overtaker']) \
      .size() \
      .reset_index(name='external_overtakes')
  ```
  即：在净超车表中，筛选 `IsTeammate == False`（超车者与被超者属于不同车队），按（年份、站次、车手）计数。
- **文献对应变量**：`External Competitive Actions`
- **文献原文定义**：
  > "We measure a rider's external competitive actions as the number of external overtakes they complete per race, i.e., the number of times they pass a rider from a different team."
- **本项目与文献的差异**：同 3.1。

---

### 3.3 `competitive_threats` — 竞争威胁数

- **中文名**：竞争威胁数（来自排名靠后车队的追赶压力）
- **单位**：个（整数，0 到当前积分榜后方车队数）
- **分析层级**：车队-赛站（constructor × round）
- **来源表**：`ALL_raw_race_results.csv` 中的 `Points` 字段逐站累加构建
- **计算逻辑**（⏳ 待 3-panel.ipynb 实现）：
  1. 按（Year, Round）顺序累加每队积分，得到每站结束时的车队积分榜
  2. 计算赛季剩余站数 × 最高单站可得分（约 26 分，含最快圈）= `max_remaining_points`
  3. `competitive_threats(t)` = 积分排名在本队**之后**、且积分差 ≤ `max_remaining_points` 的车队数
  - 即：有能力在赛季结束前追上本队积分的下位车队数量
- **文献对应变量**：`Competitive Threats`
- **文献原文定义**：
  > "A team faces a competitive threat from a lower-ranked team if the points gap between them is smaller than the maximum points a team can obtain from the remaining races."
- **本项目与文献的差异**：
  - 文献在 MotoGP 使用同类积分差逻辑，F1 积分制略有不同（最高 26 分/站含最快圈奖励）

---

### 3.4 `competitive_opportunities` — 竞争机会数

- **中文名**：竞争机会数（可追赶的上方车队数）
- **单位**：个（整数，0 到当前积分榜前方车队数）
- **分析层级**：车队-赛站（constructor × round）
- **来源表**：`ALL_raw_race_results.csv` 中的 `Points` 字段逐站累加构建
- **计算逻辑**（⏳ 待 3-panel.ipynb 实现）：
  1. 同 3.3 第 1–2 步
  2. `competitive_opportunities(t)` = 积分排名在本队**之前**、且积分差 ≤ `max_remaining_points` 的车队数
  - 即：本队仍有能力在赛季结束前追上的上位车队数量
- **文献对应变量**：`Competitive Opportunities`
- **文献原文定义**：
  > "A team has a competitive opportunity against a higher-ranked team if the points gap is smaller than the maximum points it can obtain from the remaining races."
- **本项目与文献的差异**：同 3.3。

---

### 3.5 `quali_gap_seconds` — 排位圈时差（资源禀赋代理）

- **中文名**：排位赛圈时差（相对杆位）
- **单位**：秒（float，正值表示慢于杆位）
- **分析层级**：车手-赛站（driver × round），可聚合为车队层面取最小值
- **来源表**：`ALL_raw_quali_laptimes.csv`
- **计算公式**（⏳ 待 3-panel.ipynb 实现）：
  ```python
  # 取每场排位赛的杆位圈时（BestLapTime_s 最小值）
  pole_time = df_quali.groupby(['Year', 'Round'])['BestLapTime_s'].min()
  # 每车手/车队的最快圈时 - 杆位圈时
  df_quali['quali_gap_seconds'] = df_quali['BestLapTime_s'] - pole_time
  # 车队层面取队内最快（即车队最具竞争力车手的资源代理）
  team_quali_gap = df_quali.groupby(['Year', 'Round', 'Team'])['quali_gap_seconds'].min()
  ```
- **物理含义**：值越小（越接近 0），说明该车手/车队的赛车性能越强；值越大，说明赛车相对场均竞争力越弱。用于代理文献中的"资源禀赋"（resource endowment）。
- **文献对应变量**：`Resource Endowment`（调节变量）
- **文献原文定义**：
  > "We measure resource endowments as the qualifying gap, defined as the difference between a team's fastest qualifying lap time and the fastest qualifying lap time overall."
- **本项目与文献的差异**：
  - 文献直接使用车队层面排位最快圈时差；本项目原始数据在车手层级，聚合时取队内最小值

## 4. 所有变量完整清单

### 4.1 因变量

- **`internal_overtakes`**：内部超车数 | 次 | `clean_overtakes_season` | `IsTeammate==True` 按（Year, Round, Overtaker）计数
- **`external_overtakes`**：外部超车数 | 次 | `clean_overtakes_season` | `IsTeammate==False` 按（Year, Round, Overtaker）计数

### 4.2 自变量

- **`competitive_threats`**：竞争威胁数 | 个 | `raw_race_results`（积分累加构建）| 排名靠后且积分差 ≤ 剩余可得分的车队数 ⏳ 待 3-panel.ipynb 实现
- **`competitive_opportunities`**：竞争机会数 | 个 | `raw_race_results`（积分累加构建）| 排名靠前且积分差 ≤ 剩余可得分的车队数 ⏳ 待 3-panel.ipynb 实现

### 4.3 调节变量

- **`quali_gap_seconds`**：排位圈时差（资源禀赋代理）| 秒 | `raw_quali_laptimes` | `BestLapTime_s - pole_time`，车队层面取队内最小值 ⏳ 待 3-panel.ipynb 实现

### 4.4 控制变量 — 车手层面

## 5. 超车检测与干扰过滤详细逻辑

### 5.1 总体流程（`process_one_race` 函数，Cell G）

```
输入: year, round_n
  │
  ▼
Step 1: 加载 R session（laps=True, messages=True, telemetry=False）
  │
  ▼
Step 2: 位置矩阵检测 → 生成候选超车列表 records（原始特征均为 NaN/False）
  │
  ▼
Step 3: 按需加载 telemetry（仅对候选超车涉及的 driver × lap 组合）
        → 建立 tel_store 缓存 Dict[(driver, lap_n)] → telemetry DataFrame
  │
  ▼
Step 4: 批量调用 _calc_* 函数，附加 10 维特征（D1–D10）
  │
  ▼
输出: full_overtakes DataFrame（含 Year/Round/EventName）
```

### 5.2 候选超车检测算法（Step 2）

**核心思路**：将每圈每个车手的赛场位置（Position）构建为位置矩阵，比较相邻两圈之间的位置变化，寻找发生换位的车手对。

**数据准备**：

```python
laps_base = sess.laps[[
    'LapNumber', 'Driver', 'DriverNumber', 'Team',
    'Position', 'PitInTime', 'PitOutTime', 'TrackStatus',
    'LapStartTime', 'Time', 'FastF1Generated', 'IsAccurate',
]].copy()

# 位置矩阵：行=圈号，列=车手缩写，值=该圈结束时赛场位置
lep = pivot_table(index='LapNumber', columns='Driver', values='Position')
# 上一圈位置矩阵
lsp = lep.shift(1)
```

**遍历逻辑**：

对每圈 `lap_n`（从第 2 圈开始）：
1. 取上圈位置 `ps = lsp.loc[lap_n]`，本圈位置 `pe = lep.loc[lap_n]`
2. 取两圈均有记录的车手集合 `common = ps.index ∩ pe.index`
3. 对每对 (A, B)，检查超车条件：
   - `ps[A] > ps[B]`：上圈 A 在 B 后面（位置编号更大）
   - `pe[A] < pe[B]`：本圈 A 在 B 前面（位置编号更小）
   - → 满足则判定为 A 超越了 B

**过滤条件**（以下任一成立则丢弃该超车事件）：

1. **排除首圈**：`lap_n == 1`。起跑时的位置变化不计为超车（赛车位置受发车混乱影响）
2. **排除 A 的出站效应**：A 在本圈或上圈有 `PitOutTime`（A 是从维修站出来导致的位置提升）
3. **排除 B 的进站效应**：B 在本圈或上圈有 `PitInTime`（B 是因进站离线导致的位置下降）
4. **排除 DNF 影响**：B 在 `dnf_drv` 集合中（B 因退赛导致位置丢失）
   - `dnf_drv` 判定逻辑：B 在最终成绩中状态不含 "Finished" 或 "Lap(s)"，且其最后有效 Position 为 NaN

> **注意**：SC/VSC 期间的超车不单独过滤（理论上 SC 期间不允许超车，但偶有特殊情况），通过 `LocalYellowFlag` 特征标记后在 off_track 过滤中处理。

**超车时间窗**：
- `SameLapWindowStart`：该圈内所有车手 `LapStartTime` 的最小值
- `SameLapWindowEnd`：该圈内所有车手 `Time`（圈结束时刻）的最小值

### 5.3 遥测数据缓存（Step 3）

telemetry 数据体积大，全量加载耗时长，采用**按需加载**策略：

```python
# 仅加载候选超车涉及的车手 × 圈次组合
cand_laps = set(full_ot['LapNumber'].unique())
cand_drvs = set(full_ot['Overtaker']) | set(full_ot['Overtaken'])

tel_store = {}   # Dict[(driver_abbr, lap_n)] → merged telemetry DataFrame
for drv in cand_drvs:
    for ln in cand_laps:
        car  = lap_row.get_car_data().add_distance().add_relative_distance()
        pos  = lap_row.get_pos_data()
        tel_store[(drv, ln)] = car.merge_channels(pos, frequency=TELEMETRY_HZ)
```

`tel_store` 中每条记录包含：`SessionTime`, `Speed`, `Throttle`, `Brake`, `DRS`, `Distance`, `RelativeDistance`, `X`, `Y`, `Z`, `Status` 等列。

### 5.4 十维特征详细说明（D1–D10）

#### D1 — `Is_DRS_Active`

- **计算对象**：超车发起者 A
- **输入**：`tel_store[(A, lap_n)]['DRS']`
- **返回类型**：`bool`（若遥测缺失返回 `np.nan`）
- **计算逻辑**：DRS 列取值含义：`0`=不可用，`8`=可用但未开启，`10`=DRS 开启。该圈内任意时刻 `DRS == 10` 则返回 `True`。
- **物理含义**：超车时是否使用了 DRS（可变后翼，降低阻力提升直道速度），用于区分 DRS 辅助超车和真正竞争性超车。

#### D2 — `RelativeDistance`

- **计算对象**：超车发起者 A
- **输入**：`tel_store[(A, lap_n)]['RelativeDistance']`
- **返回类型**：`float`（0–1，若遥测缺失返回 `np.nan`）
- **计算逻辑**：取 A 该圈 `RelativeDistance`（赛道完成比例）的中位数，作为该圈的"代表位置"
- **物理含义**：该超车事件大致发生在赛道的哪个位置（0=发车线，1=一圈结束）。辅助数据

#### D3 — `Throttle_Drop_Index`（油门收油指数）

- **计算对象**：超车发起者 A 和被超车手 B
- **输入**：`tel_store[(A, lap_n)]['Throttle']`，`tel_store[(B, lap_n)]['Throttle']`
- **返回类型**：`float`（若段内数据 < 5 帧返回 `np.nan`）
- **计算逻辑**：
  1. 将 A 和 B 的遥测数据按 `SessionTime` 对齐（`pd.merge_asof`，方向 `nearest`）
  2. 截取赛道 40%–80% 位置段（关键加速区）：`0.4 ≤ RelativeDistance ≤ 0.8`
  3. 计算 `median(|A.Throttle - B.Throttle|)`
- **物理含义**：在关键加速段，A 和 B 的油门开度差异的中位数。值越大，表示 B 相对 A 收油越明显，疑似主动让位（车队指令）。

#### D4 — `Brake_Pressure_Delta`（刹车时序差，ms）

- **计算对象**：超车发起者 A 和被超车手 B
- **输入**：`tel_store[(A/B, lap_n)]['Brake']`，`tel_store[(A/B, lap_n)]['SessionTime']`
- **返回类型**：`float`（ms，若可配对事件不足返回 `np.nan`）
- **计算逻辑**：
  1. 分别提取 A 和 B 的刹车上升沿事件（`Brake: 0 → 1`）的时间戳列表 `ea`、`eb`
  2. 对 `ea` 中每个事件，在 `eb` 中找最近的配对事件
  3. 仅保留时间差 < 500ms 的配对，取所有配对差值的中位数（单位 ms）
- **物理含义**：两车刹车时序的整体差异。值越大，表示 B 比 A 早刹车越多（或晚刹车），疑似有意识地调整驾驶节奏（车队指令让位）。

#### D5 — `TeamOrderConfidence`（车队指令置信度）

- **计算对象**：综合 D3 和 D4
- **输入**：`tdi`（D3 结果），`bpd`（D4 结果），`TEAM_ORDER_THROTTLE_THRESHOLD`，`TEAM_ORDER_BRAKE_THRESHOLD`
- **返回类型**：`float`（0–1，若两者均为 NaN 返回 `np.nan`）
- **计算公式**：
  ```python
  score = min(tdi / 10.0, 1.0) × 0.6   # 油门收油贡献 60% 权重
         + min(bpd / 150.0, 1.0) × 0.4  # 刹车时序贡献 40% 权重
  ```
- **物理含义**：综合油门和刹车行为，评估本次超车为车队指令让位的可能性。≥ 0.5 则标记为 `team_order`。

#### D6 — `Vmin_Deviation`（最低速偏差，%）

- **计算对象**：被超车手 B
- **输入**：`tel_store[(B, lap_n)]['Speed']`，`laps_base`（历史参考圈）
- **返回类型**：`float`（%，若参考圈或数据不足返回 `np.nan`）
- **计算逻辑**：
  1. 检查本圈是否为黄旗圈（`TrackStatus` 含 `'2'`）；若否返回 `np.nan`
  2. 找 B 的参考圈：`TrackStatus == '1'`（绿旗）且 `IsAccurate == True` 的圈，取圈号中位数
  3. 在赛道后段（`RelativeDistance > 0.7`，即刹车区域）分别取本圈和参考圈的最低速
  4. `Vmin_Deviation = (current_min - ref_min) / ref_min × 100`
- **物理含义**：黄旗圈内 B 在刹车区的最低速相对正常圈的偏差。负值（速度更低）说明 B 在刹车区显著减速，疑似方向感失误或越界。

#### D7 — `OffTrack_Flag`

- **计算对象**：被超车手 B
- **输入**：`tel_store[(B, lap_n)]['Status']`
- **返回类型**：`bool`（若遥测缺失返回 `False`）
- **计算逻辑**：该圈内 `Status == 'OffTrack'` 的数据帧占比 > 5% 则返回 `True`
- **物理含义**：B 在该圈内有明显的出赛道行为（冲出赛道），可能导致被动让位。

#### D8 — `Lockup_Flag`（锁死标志）

- **计算对象**：被超车手 B
- **输入**：`tel_store[(B, lap_n)]['Brake']`，`tel_store[(B, lap_n)]['Throttle']`
- **返回类型**：`bool`（若遥测缺失返回 `False`）
- **计算逻辑**：满足 `Brake == 1`（全力刹车）且同时 `Throttle > 30`（油门未完全收回）的数据帧 ≥ 2 帧
- **物理含义**：检测 B 的刹车锁死行为（刹车时仍有油门，典型的刹车点估算失误特征），说明该超车可能源于 B 的操作失误而非竞争压力。

#### D9 — `LocalYellowFlag`（局部黄旗）

- **计算对象**：全场（圈级别）
- **输入**：`laps_base['TrackStatus']`
- **返回类型**：`bool`
- **计算逻辑**：该圈内任意车手的 `TrackStatus` 含字符 `'2'` 则返回 `True`
- **物理含义**：超车发生圈次是否有黄旗。黄旗期间技术上禁止超车，此时发生的换位更可能源于一方减速避让而非真实竞争。

#### D10 — `OffTrackConfidence`（越界/干扰置信度）

- **计算对象**：综合 D6–D9
- **输入**：`vmin_dev`（D6），`offtrack`（D7），`lockup`（D8），`yellow`（D9）
- **返回类型**：`float`（0–1，四项均未触发返回 `np.nan`）
- **计算公式**：
  ```python
  score = 0.4 × (vmin_dev > 5.0)    # 速度偏差贡献 40% 权重
         + 0.3 × offtrack            # 出赛道贡献 30% 权重
         + 0.2 × lockup              # 锁死贡献 20% 权重
         + 0.1 × yellow              # 黄旗贡献 10% 权重
  ```
- **物理含义**：综合多种驾驶行为异常指标，评估本次超车为非正常竞争性超车（越界/锁死/黄旗）的可能性。≥ 0.5 则标记为 `off_track`。

### 5.5 净超车生成逻辑（Cell H）

```python
# 1. 标记 vehicle_order 记录（四元组键）
dirty_team = set(zip(yr_team['Round'], yr_team['LapNumber'],
                     yr_team['Overtaker'], yr_team['Overtaken']))
dirty_off  = set(zip(yr_offtrack['Round'], yr_offtrack['LapNumber'],
                     yr_offtrack['Overtaker'], yr_offtrack['Overtaken']))

# 2. 并集为"脏"超车
dirty = dirty_team | dirty_off

# 3. 从 full 中排除脏记录，得到 clean
mask_c = ~pd.Series(
    list(zip(yr_full['Round'], yr_full['LapNumber'],
             yr_full['Overtaker'], yr_full['Overtaken'])),
    index=yr_full.index
).isin(dirty)
yr_clean = yr_full[mask_c].copy()
```

> **唯一键**：`(Round, LapNumber, Overtaker, Overtaken)` 四元组。同一场同一圈同一对车手只可能有一条记录（超车检测算法保证唯一性）。

## 6. 数据局限与注意事项

### 6.1 CIRCUIT_META 硬编码的局限

- `CIRCUIT_META` 以 **2023 赛季**的赛道配置为基准，部分赛道历年有变化：
  - 澳大利亚（墨尔本）：2022 赛道重新设计后 DRS 区由 3 增至 4
  - 新加坡（海湾赛道）：2023 赛道缩短并调整 DRS 区
  - 部分街道赛道每年 DRS 激活区段微调
- **影响**：`DRS_Zones` 字段在 2018–2022 的部分站次可能存在 ±1 的偏差
- **建议**：如对 DRS 区数敏感，可在 3-panel.ipynb 中针对已知变更赛道手动修正

### 6.2 FastF1 数据可用性

- FastF1 官方支持的历史数据下限为 **2018 赛季**（2018 F1 澳大利亚 GP 起）
- 2018–2019 赛季部分场次遥测数据可能不完整或缺失特定字段
- `sess.total_laps` 属性在早期赛季可能返回 `None`，已通过 `getattr(sess, 'total_laps', np.nan)` 处理

### 6.3 2026 赛季数据缺失

- 截至 **2026-03-27**，2026 赛季仅完成前 2 站（澳大利亚 GP + 中国 GP）
- Round 3+ 比赛尚未举行，对应数据文件不存在
- 处理方式：`1-other.ipynb` 中通过 `MAX_2026_ROUNDS = 2` 控制，`1-overtakes.ipynb` 通过赛历自动感知已完成站次
- 赛季结束后：将 `MAX_2026_ROUNDS` 调大到实际总站数，重跑 Cell 3b 和 Cell H 即可补全

### 6.4 TrackStatus 的聚合留待 3-panel.ipynb

- `raw_track_status` 中 `TrackStatus` 保留**原始字符串格式**，未做聚合
- 下游在 3-panel.ipynb 中按站次聚合为 `sc_lap_fraction`、`vsc_lap_fraction`、`yellow_lap_fraction` 等比例变量
- 注意 `TrackStatus` 可能为复合字符串（如 `'24'`），聚合时需用 `str.contains` 而非等值匹配

### 6.5 Sprint 赛程的排位赛口径

- Sprint 冲刺赛周末存在两场排位赛：
  - `Q`：主排位赛（决定正赛发车顺序）
  - `SQ`：冲刺资格赛（决定冲刺赛发车顺序）
- 本项目统一使用 `Q`，保持与常规赛周的数据口径一致
- 这意味着 Sprint 赛周的 `raw_quali_laptimes` 反映的是主排位赛而非冲刺赛的速度表现

### 6.6 多个 Location 指向同一赛道

| 时间范围 | 2004–2020（17 赛季） | 2018–2026（9 赛季） |
| 车队规模 | 通常每队 2 名车手（工厂队/卫星队结构） | 每队固定 2 名车手 |
| 赛道特征 | 无 DRS 机制 | 有 DRS（可变后翼），影响超车动力学 |
| 积分制 | MotoGP 积分制（最高 25 分/站） | F1 积分制（最高 26 分/站含最快圈） |
| 资源禀赋代理 | qualifying gap（与本项目一致） | qualifying gap（`BestLapTime_s - pole_time`） |
| 超车定义 | 同类（赛场位置变化） | 同类（赛场位置变化，通过 FastF1 圈速数据检测） |
| 数据来源 | 官方 MotoGP 数据 | FastF1 API（官方 F1 数据） |

### 6.8 超车检测算法的潜在误差来源

- **圈级别精度**：超车时间戳精确到圈，无法定位圈内具体时刻（`SameLapWindowStart/End` 为时间窗，非精确时刻）
- **FastF1Generated 圈**：FastF1 可能自动补全某些缺失圈数据（`FastF1Generated == True`），这些圈的 Position 可能不准确，但设计上未单独过滤（后续可考虑加强过滤）
- **同圈多次换位**：如两名车手在同一圈内换位超过一次，算法只记录该圈结束时的最终净变化，中间过程不记录
- **安全车期间**：SC/VSC 进坑导致的虚拟换位已通过 PitIn/PitOut 过滤，但 SC 期间某些边缘情况可能漏进候选集，通过 `LocalYellowFlag` 和 `Vmin_Deviation` 特征识别后在 off_track 表中剔除
